# RAS Test Case Validator

**Purpose:** Validate executed RAS OAuth workflow test cases against defined validation criteria. Parses test case specifications, compares execution results with pass/fail requirements, validates artifact integrity (logs and screenshots), and generates structured test execution reports.

**Input:** Test case identifier (`Case_ID`+`TestName` combined), Excel test case workbook, execution log artifacts  
**Output:** Test execution report (JSON and HTML) with pass/fail verdict, validation evidence, and remediation guidance  
**Scope:** RAS Testing workspace — post-execution validation for CTDC and C3DC test suites

---

## What It Does

This skill automates validation of executed RAS test cases by performing the following operations:

1. **Parsing Test Case Definition** — Reads Excel workbook to extract:
   - `Case_ID` — Test/user identifier. **Not guaranteed unique** — often repeats across rows for different scenarios of the same user.
   - `TestName` — Human-readable scenario name for the row; combine with `Case_ID` (slugified, e.g. `<Case_ID>_<TestName>`) to get the unique test identifier used to find execution artifacts and name reports.
   - `Validate` — Pass/fail criteria in natural language, specifying conditions for success and failure and how to validate execution results.
   - `Test Steps` — Expected workflow steps executed
   - `System` and `User` — Settings used for execution

2. **Locating Execution Artifacts** — Discovers generated files, keyed by the combined `<Case_ID>_<TestName>` identifier (referred to below as `<testname>`):
   - JSON log: `test-results/workflow-logs/<testname>-<timestamp>.json`
   - Text log: `test-results/workflow-logs/<testname>-<timestamp>.log`
   - Screenshots: `test-results/screenshots/<timestamp>_*.png` (matching the `<testname>` run)

3. **Validating Execution Results** — Compares actual results against criteria:
   - **Authorization Success:** Authorization code captured, callback URL valid
   - **Token Exchange:** Access token, refresh token, ID token received (HTTP 200)
   - **User Info Retrieval:** Subject claim present, Passport claim available (HTTP 200)
   - **Screenshots Captured:** All expected step screenshots present
   - **No Errors:** All steps completed without assertion failures

4. **Generating Validation Report** — Produces artifacts:
   - JSON report: Structured pass/fail results, evidence links, error details
   - HTML report: Visual dashboard generated by `generate-dashboard.mjs` with step-by-step validation, screenshots, and logs
   - Plain text summary: Quick executive summary

## Excel Test Case Format

### Required Columns

| Column | Type | Purpose | Example |
|--------|------|---------|---------|
| Case_ID | String | User/test identifier; **may repeat** across rows | `yizhen.chen_id_green_file` |
| TestName | String | Scenario name for this row; combined with `Case_ID` to form the unique test identifier | `Get Authenticated` |
| Test Steps | String | Workflow steps executed | `authorize/login through RAS, exchange code, call userinfo` |
| Validate | String | Pass/fail criteria (natural language) | `Pass: User successfully authorized and userinfo retrieved. Fail: otherwise` |
| System | String | Shared RAS configuration path | `./CTDC/system.env` |
| User | String | User-specific credentials path | `/CTDC/someone.env` |

**Test identifier used to locate artifacts:** `<Case_ID>_<TestName>` (slugified), not `Case_ID` alone.

### Validation Criteria Syntax

The `Validate` column supports natural language descriptions with **Pass** and **Fail** sections and how to validate the results of execution.

```
Pass: [conditions for success]
Fail: [conditions for failure / error states]
```

**Examples:**

- `Pass: Authorization code captured and callback received. Fail: Missing authorization code or callback timeout.`
- `Pass: All three tokens received (access, refresh, ID). Fail: Missing any token or HTTP error.`
- `Pass: User info retrieved with valid subject and Passport claim. Fail: Missing claim or user info call failed.`
- `Pass: All screenshots captured at each step. Fail: Missing screenshots or step execution incomplete.`

## Validation Workflow

### Step 1: Accept Test Case Identifier
User provides test case name (combined `Case_ID`+`TestName`) and optional workbook path.

### Step 2: Parse Test Case Definition
Read Excel workbook to extract validation criteria and expected steps from `Validate` column, matching the row by its combined `Case_ID`+`TestName` identifier.

### Step 3: Locate Execution Artifacts
Search `test-results/workflow-logs/` for JSON and text logs matching the combined `<Case_ID>_<TestName>` identifier.
Search `test-results/screenshots/` for PNG files from the same execution run.

### Step 4: Parse Execution Results
Read JSON log file and extract:
- Overall execution status (passed/failed)
- Individual step results (duration, HTTP status, captured fields)
- Error messages (if any)

### Step 5: Validate Against Criteria
Compare execution results with validation criteria:

**Authorization Step Validation:**
- ✓ Authorization code captured (`capturedCode: true`)
- ✓ Callback URL matches expected origin
- ✓ HTTP status indicates success (no timeouts or redirects)

**Token Exchange Validation:**
- ✓ HTTP 200 response received
- ✓ Access token present and non-empty
- ✓ Refresh token present and non-empty
- ✓ ID token present and non-empty

**User Info Validation:**
- ✓ HTTP 200 response received
- ✓ Subject claim present and non-empty
- ✓ Passport claim present (JWT or array format)

**Consent Group Validation (step 12, `get-consent-groups`):**
- ✓ `consentGroupCount` > 0 and `consentGroups` array populated from the passport's second visa (`ras_dbgap_permissions`)
- ✓ For consent-code/controlled-data scenarios, cross-check the DRS-accessed object's `phs_id`/`consent_group` (if exposed in the DRS response) against `context.data.consentGroups` to confirm the accessed file actually belongs to an authorized consent group — do not assume a DRS 200 response alone proves correct consent-based authorization
- ~ If the DRS response never exposes a `phs_id`/`consent_group` identifier to compare against (as observed with plain public-object test data), note this as a validation gap rather than a pass/fail — a technical 200 does not confirm consent-group correctness

**Screenshot Validation:**
- ✓ Minimum expected screenshots captured (e.g., 10+ for id.me, 12+ for login.gov)
- ✓ Screenshot files exist and are readable
- ✓ Screenshot timestamps align with execution run
- ✓ The screenshot contains the correct visual content and matches the expected state of the application at the time it was captured. If the AI model being used does not have vision capabilities, inform the user that the screenshot must be verified manually.


### Step 6: Write Validation Results
Write the structured JSON validation report before generating the final dashboard. Include:
- Test case metadata (ID, steps, execution time)
- Validation verdict (PASS / FAIL / PARTIAL)
- Per-step validation details with evidence
- Screenshots with annotations
- Error details and remediation guidance

### Step 7: Generate and Deliver the Final Dashboard
From the RAS Testing workspace root, run the repository dashboard generator after the JSON validation report has been saved:

```bash
node generate-dashboard.mjs
```

`npm run dashboard:generate` is an equivalent repository command. Do not handcraft a separate final HTML report. Confirm that the generator created:

```text
test-results/dashboard/index.html
```

In the final response:

1. State the validation verdict and concise summary.
2. Provide the absolute local path to `test-results/dashboard/index.html` as the final report location.
3. If an in-app browser is available, open that generated `index.html` and show it to the user; still provide the absolute local path so the report can be reopened later.
4. If dashboard generation fails, report the error and do not claim that the final report was generated.

## Validation Rules

### Pass Conditions
Test case is marked **PASS** when:
- ✓ All executed steps returned expected results
- ✓ HTTP status codes match expected values (200 for token/userinfo endpoints)
- ✓ All required tokens/claims are present and non-empty
- ✓ Authorization code captured and callback URL valid
- ✓ All screenshots exist and match execution timestamps
- ✓ No assertion errors in step execution

### Fail Conditions
Test case is marked **FAIL** when:
- ✗ Any step execution failed (non-zero exit code)
- ✗ HTTP error status received (4xx, 5xx)
- ✗ Required token or claim is missing or empty
- ✗ Authorization code not captured or callback invalid
- ✗ Screenshots missing for one or more executed steps
- ✗ Timeout during browser automation or API call
- ✗ Validation criteria explicitly not met

### Partial Pass
Test case is marked **PARTIAL** when:
- ~ Some required artifacts missing but core authentication succeeded
- ~ Optional screens (e.g., consent) not present but workflow continued
- ~ Screenshots partially captured (most but not all steps)

## Error Handling

### Missing Execution Log
**Issue:** No JSON log file found matching the combined `Case_ID`+`TestName` identifier  
**Action:** Report missing artifact; suggest running the test case first using RAS Test Case Executor skill

### Missing Screenshots
**Issue:** Expected screenshots not found in test-results/screenshots/  
**Action:** Report partial validation; note which steps are missing visual evidence; mark as PARTIAL

### Invalid Validation Criteria
**Issue:** `Validate` column has invalid or missing syntax  
**Action:** Report parsing error; provide template for valid criteria format

### Malformed JSON Log
**Issue:** Execution log JSON is corrupted or incomplete  
**Action:** Report parse error; attempt to use text log as fallback; mark validation as incomplete

### Ambiguous Pass/Fail Criteria
**Issue:** Validation criteria is vague or conflicting (e.g., multiple pass conditions without clear AND/OR logic)  
**Action:** Report interpretation; validate against most common success criteria; flag ambiguity in report
